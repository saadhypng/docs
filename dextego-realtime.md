# Realtime Coaching — System Design Spec

> Status: Draft 1 · Scope: Phase 1 (Recall.ai-backed, Zoom / Google Meet / MS Teams)
> Owner: TBD · Related: `notes/architecture.md` (pending), `notes/features/realtime-coaching.md` (to create on implementation)

---

## 1. Context & Motivation

Dextego today is a **post-call** coaching product:

- Calendar events ingested via Nango (Google Calendar, Outlook).
- Meeting briefs generated by Mastra agents on a schedule (pre-call).
- Pitch scoring and buyer personality insights generated after a roleplay/session via Inngest.
- Streaming chat UI uses SSE (`/api/chat` + `sse.ts`, consumed by `useChatStream`).
- Vector search backed by Supabase pgvector (`kb_chunks`, 1536-dim, `text-embedding-3-small`).

We want to move Dextego **into the call**. Reps on Zoom / Meet / Teams should get live guidance from Tego — suggested responses to buyer questions, framework tracking (MEDDIC, SPICED, etc.), risk flags, and clarifying prompts — while the call is happening. After the call we reuse the existing post-call pipeline, fed by the real transcript.

Peer reference: **insyghtful.ai** (live call guidance + speaker-ID transcripts + automated post-call CRM insights). Our differentiation: Mastra-agent coaching depth, framework-driven artifacts, buyer personality reveal grounded in calendar context, and KB RAG.

This spec is **high-level system design** (per original ask) but detailed enough to execute against — it names components, data contracts, events, critical files, and sequencing.

## 2. Goals & Non-Goals

### Goals (Phase 1)

1. Schedule a Recall.ai bot into Zoom / Meet / Teams calls based on user settings and calendar events.
2. Two session modes:
   - **Recording-only** — bot joins, records, we process only post-call.
   - **Realtime coaching** — bot joins with realtime transcript enabled; Tego consumes it live; rep sees a live session in Dextego.
3. Speaker identification — map Recall diarization labels onto calendar attendees with role (host/buyer/other).
4. Live session UI — active-session indicator on dashboard, click-through to live view with transcript, speaker tiles, Tego suggestions, framework artifacts, and free-form ask bar.
5. Post-call finalization — store recording, persist full transcript, vectorize into pgvector, trigger existing brief + pitch score + buyer personality jobs.
6. Observability — Langfuse tracing for realtime LLM pipeline; operational metrics for bot lifecycle.
7. Gating — org plan entitlement + per-user toggle.

### Non-Goals (Phase 1)

- Native Zoom / Meet / Teams SDK integrations (Recall abstracts all three).
- Two-way bot interaction (bot speaking back, chat posting).
- Realtime video rendering from Recall (placeholder tiles only).
- CRM write-back during the call.
- Non-English languages (English first; i18n pass after Phase 1).
- Mobile live view (desktop-only).

---

## 3. User Stories

1. **Rep with realtime enabled** schedules a discovery call in Google Calendar → Dextego auto-schedules a bot → at call start the rep clicks the "Live session" chip in Dextego → sees transcript + Tego suggestions as the call progresses → hangs up → finds brief + pitch score + buyer profile within ~2 min.
2. **Rep on recording-only tier** — same, minus the live view and realtime suggestions; post-call deliverables are identical.
3. **Rep on free plan / no entitlement** — no bot is scheduled; no regression in existing product.
4. **Admin / user settings** — a rep toggles realtime off for a sensitive meeting; bot still records (if allowed) but no live feed or Tego.

## 5. Key Decisions (confirmed)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Client transport | **SSE** | Matches existing Dextego streaming; unidirectional is enough for Phase 1 (user sends free-form asks via separate POST to `/api/chat`-style endpoint). |
| Internal fanout bus | **Supabase Realtime broadcast** | Already in stack; avoids standing up new infra. Gateway publishes; SSE route subscribes. |
| Realtime LLM | **GPT-4o + GPT-4o-mini tiered** | 4o-mini for cheap classification on every finalized utterance; 4o for answer synthesis only on triggers. |
| Embeddings | **text-embedding-3-small (1536-dim)** | Consistent with `kb_chunks`; allows shared retrieval across KB + transcripts if desired. |
| Gating | **Org plan entitlement + per-user toggle** | Plan controls availability; user decides per-meeting behavior via settings. |
| Video provider abstraction | **Recall.ai only (Phase 1)** | Single integration covers all three platforms. |

## 6. Data Model (conceptual)

> DDL deferred to migration; names and relationships below.

### 6.1 New tables

- **`meeting_sessions`**
  - `id` (uuid, PK)
  - `org_id`, `user_id`, `calendar_event_id`, `meeting_brief_id` (nullable; linked if brief exists)
  - `recall_bot_id` (nullable until scheduled)
  - `mode` enum: `recording` | `realtime`
  - `status` enum: `scheduled` | `joining` | `live` | `processing` | `completed` | `failed` | `cancelled`
  - `meeting_url`, `platform` enum: `zoom` | `meet` | `teams`
  - `scheduled_start`, `started_at`, `ended_at`
  - `recording_url`, `recording_storage_path`
  - `error` JSONB
  - RLS: user can read own; org admins can read org-scoped.

- **`session_transcripts`**
  - `id`, `session_id` (FK), `speaker_label`, `speaker_id` (nullable, FK to `session_speakers`)
  - `ts_start`, `ts_end` (ms from call start)
  - `text`, `words` JSONB (word-level with timings), `is_final` bool
  - Partition by `session_id` or `created_at` (month) for growth.

- **`session_speakers`**
  - `id`, `session_id`, `speaker_label` (Recall), `attendee_email` (nullable), `display_name`, `role` enum: `host` | `rep` | `buyer` | `other`
  - `confidence` float, `locked` bool

- **`session_tego_events`**
  - `id`, `session_id`, `type` enum (`suggestion` | `framework_update` | `risk_flag` | `clarifying_question` | `classification`)
  - `payload` JSONB (suggestion text, framework slot, rationale, citations)
  - `utterance_anchor_ids` array — which utterances triggered this
  - `latency_ms`, `model`, `tokens_in`, `tokens_out`, `cost_usd`

- **`transcript_chunks`** (pgvector, mirrors `kb_chunks`)
  - `id`, `session_id`, `org_id`, `user_id`
  - `chunk_text`, `embedding` vector(1536)
  - `metadata` JSONB (speaker, ts_start, ts_end, role)
  - Index: ivfflat or hnsw on `embedding`.

- **`user_realtime_settings`** (or extend existing user settings table)
  - `user_id` PK
  - `realtime_enabled` bool
  - `bot_join_behavior` enum: `all` | `external_only` | `off`
  - `default_framework` enum (MEDDIC / SPICED / BANT / custom)
  - `language` enum (en initially)
  - `recording_consent_ack` bool + `consent_ack_at`

### 6.2 Reused

- `meeting_briefs` — gains a pointer via `meeting_sessions.meeting_brief_id`.
- `pitch_scores` — produced by existing scorer but now keyed to `session_id` (new nullable FK).
- `buyer_personality_profiles` — fed by finalizer.

## 7. External Contracts

### 7.1 Recall.ai

- **`POST /api/v1/bot`** — schedule bot. Key params: `meeting_url`, `bot_name`, `transcription_options: { provider: "deepgram" | ... }`, `real_time_transcription: { destination_url, webhook_auth }` when realtime; `recording_mode`.
- **Webhooks** (→ `/api/recall/webhook`):
  - `bot.status_change` — `joining_call` → `in_call` → `call_ended` → `done`.
  - `transcript.data` (fallback if we don't use WS).
  - `recording.done` — recording URL available.
- **Realtime WS** — utterance stream: `{ speaker, words: [{ text, start, end, confidence }], is_final }`.
- Security: HMAC signature on webhooks; shared secret env var; reject non-matching.

### 7.2 Internal Event Bus (Supabase Realtime broadcast)

Channel naming: `session:{session_id}`.

Events (topic → payload):

| Event | Source | Payload |
|-------|--------|---------|
| `transcript.utterance` | Gateway | `{ utterance_id, speaker_label, text, ts_start, ts_end, is_final }` |
| `speaker.update` | Speaker ID | `{ speaker_label, attendee_email, display_name, role, confidence, locked }` |
| `tego.suggestion` | Tego Engine | `{ id, title, body, rationale, citations[], anchor_utterance_ids[] }` |
| `tego.framework` | Tego Engine | `{ framework, slots: [{ key, status, evidence_utterance_ids[] }] }` |
| `tego.risk` | Tego Engine | `{ severity, message, anchor_utterance_ids[] }` |
| `tego.clarifying_question` | Tego Engine | `{ question, reason }` |
| `session.status` | Scheduler/Finalizer | `{ status, error? }` |

### 7.3 Client SSE Contract

`GET /api/sessions/:id/stream` — text/event-stream.

- Auth: PropelAuth session; server validates user owns the session or is org admin.
- Events mirror the internal bus 1:1. Server adds `event: <topic>` and `data: <json>`.
- Heartbeat every 15s to keep connections open through Vercel.
- Client reconnects with `Last-Event-ID` for at-least-once replay.

### 7.4 Free-form "Ask Tego" during a call

- `POST /api/sessions/:id/ask` — body `{ question }` → streams response via the same SSE channel (`tego.suggestion` with `source: "user_ask"`).

## 8. Sequencing / End-to-End Flows

### 8.1 Bot Scheduling

1. Nango ingest creates/updates a `meeting_briefs` row (existing).
2. Inngest `meetingSessionsScheduler` runs every N minutes: finds meetings starting in the next 15 min with `meeting_url`, where owner user has realtime entitlement + setting.
3. For each, calls Recall `POST /bot`. Creates `meeting_sessions` row with `mode`, `status=scheduled`, `recall_bot_id`.
4. User-setting changes or calendar deletions → cancel bot via `DELETE /bot/:id`.

### 8.2 Call Start (realtime mode)

1. Recall posts `bot.status_change=in_call` → webhook handler updates `meeting_sessions.status=live`, enqueues `session.live` event.
2. Gateway function spins up (or an already-running pool worker picks it up), opens Recall WS for that bot, starts Speaker ID.
3. Dashboard sees session become `live` via Supabase Realtime and surfaces the chip.

### 8.3 During Call — Realtime Pipeline

1. Utterance arrives on WS → Gateway persists (`session_transcripts`) and publishes `transcript.utterance`.
2. Speaker ID compares speaker labels vs calendar attendees + heuristics (name match, domain of email, sequence of first-speakers). Emits `speaker.update` on changes.
3. Tego Engine subscribes to finalized utterances:
   - **Classifier (GPT-4o-mini)**: inputs last N utterances (sliding window, e.g. 20 turns) + current framework state. Output: `{ is_trigger, trigger_type, confidence }`. Debounce: minimum 8s between synthesizer calls; rate-limit per session.
   - **Synthesizer (GPT-4o)** on trigger: inputs trigger + wider context + KB RAG results (`kbVectorQuery`) + buyer profile + user framework. Output: suggestion JSON. Persist to `session_tego_events`, publish `tego.suggestion`.
   - Framework tracker updates slot states, publishes `tego.framework` when changed.
4. Live UI renders events as they arrive.

### 8.4 During Call — Recording-only mode

- Gateway still persists the transcript (if available via WS or via Recall partial transcripts) but no Tego subscription; no live UI rendered beyond status card.

### 8.5 Call End

1. Recall `bot.status_change=call_ended` → session → `processing`.
2. Recall `recording.done` → Finalizer Inngest fn:
   - Download recording → upload to Supabase Storage (bucket: `session-recordings`, path `{org_id}/{session_id}.mp4`); store path + signed-URL policy.
   - Pull full transcript (Recall REST) if not already complete; upsert final utterances.
   - Enqueue **Vectorizer** (chunk ~400 tokens, overlap 60; embed `text-embedding-3-small`; upsert `transcript_chunks`).
   - Enqueue existing **meetingBrief**, **pitchScoreGeneration**, and **buyerPersonalityReveal** jobs, passing `session_id`.
3. On success, `session.status=completed`; UI links to post-call review.

### 8.6 Failure Modes

- Bot fails to join → `status=failed`, user notified in dashboard + optional email.
- Gateway WS drops → auto-reconnect with backoff; fallback to webhook transcript polling.
- Recall recording.done not received within 30 min of `call_ended` → reconciler job pulls status.
- LLM rate-limit → classifier/synthesizer retry with jitter; drop non-critical suggestions if queue backlog >N.

## 9. Tego Realtime Engine — Detail

### 9.1 Pipeline

```
 utterance (final) ──▶ windowBuffer ──▶ classifier (4o-mini)
                                                │
                                  no trigger ◀──┤──▶ trigger
                                                │
                                                ▼
                           retriever (kbVectorQuery + transcript_chunks)
                                                │
                                                ▼
                           synthesizer (4o)  ──▶ emit tego.suggestion
```

### 9.2 Prompts (high-level)

- **Classifier system prompt**: identify buyer question, objection, framework-slot signal, stall/silence, competitive mention, pricing request. Output strict JSON.
- **Synthesizer system prompt**: given trigger + context + framework + KB citations, produce a concise suggested response (≤60 words), optional talk-track, and rationale. Must cite sources (`citations[]`) where grounded by KB.
- **Framework tracker**: separate lightweight call (can run on 4o-mini) that re-evaluates slot completion every M utterances.

### 9.3 Cost / Rate Controls

- Max 1 synthesizer call per 8s per session.
- Max 20 synthesizer calls per 30-min call (configurable by plan).
- Classifier always runs on finalized utterances; batched every 2s.
- Token caps enforced; on overflow, summarize sliding window via a cheap pass.

### 9.4 Observability

- Langfuse spans: one trace per utterance, child spans for classifier/retriever/synthesizer.
- Metrics: latency p50/p95 end-to-end (utterance-final → UI event), trigger rate, suggestion acceptance (future: user rating).
- Per-session cost recorded on `session_tego_events`.

## 10. Speaker Identification — Detail

### 10.1 Inputs

- Recall diarization: `speaker_label` (e.g., `Speaker 0`), plus (platform-dependent) participant metadata (display name, email when available).
- Calendar attendees from `meeting_briefs.participants` / `external_participants`.

### 10.2 Heuristics

1. **Direct match** — Recall participant display name/email ↔ calendar attendee.
2. **Self-intro** — first 60s: scan for "I'm X / This is X / X here"; map to attendee name similarity (Jaro-Winkler ≥ 0.85).
3. **Host detection** — bot host / organizer email → `role=host`.
4. **Domain split** — host-org domain = `rep`, other = `buyer`, else `other`.
5. **Confidence** = weighted sum; lock at ≥0.75 or after 2 min.

### 10.3 Output

- `session_speakers` rows; `speaker.update` events emitted on changes.
- Used by UI to label transcript lines and tiles, and by Tego for role-aware prompts.

- Accessibility: transcript as list with role attributes; keyboard shortcut to copy last suggestion.

## 13. Security, Privacy, Compliance

- **Consent** — user must ack recording/AI participation policy before first bot schedule. Bot display name includes "Dextego AI Notetaker". Where required by jurisdiction, require explicit attendee consent (config per org).
- **PII** — transcripts treated as sensitive. Storage encrypted at rest (Supabase default). Signed URLs for recording access, short-lived.
- **RLS** — all new tables org/user-scoped with RLS policies.
- **Data retention** — configurable per org (default 365 days transcripts, 90 days recordings, then purge).
- **Vector store isolation** — `transcript_chunks` queries always filter by `org_id` (and by user where needed) at retrieval time; enforced in the tool wrapper, not only by RLS.
- **Webhook verification** — Recall HMAC check; reject on mismatch.
- **SSE auth** — PropelAuth session cookie; session-level ownership check.

## 15. Performance & Cost Targets

- Utterance → live UI render: p95 < 1.5s.
- Utterance (final) → Tego suggestion render: p95 < 5s end-to-end.
- Cost target per 30-min realtime call: < $0.75 (model + infra) at current OpenAI pricing; cap synthesizer calls accordingly.
- Vectorizer finishes within 2 min of `recording.done` for a 60-min call.
- Recording file < 500MB for 1 hour (use Recall's compressed mp4).

## 19. Open Questions / Risks
